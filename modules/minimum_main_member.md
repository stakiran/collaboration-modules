# Minimum Main Member Principle

## 概要
**Minimum Main Member Principle** とは、メインとなるメンバーを最小限に留めましょうという原則であり、以下を定義する:

- メインとは「主たる仕事として請け負っている」ことと「意思決定権を持つこと」の **両方** が当てはまること
    - 文化的にどうしても厳しい場合は片方で構いません
- 最小限とはできるだけ少ないこと、しかし上限でも自分を入れて 4 人以下であること

## 3M Principle のパターン例
パターンは各自つくれば良いし、組織の数だけ色があるため、おそらくつくった方が良い。

### 作業者とレビュアーパターン
2人

- 作業者：つくる
- レビュアー：つくったものをレビューする

### エンジニアリング、デザイン、マネジメントパターン
3人

- エンジニア：つくる
- デザイナー：UI、UX、その他ビジュアルを担当する
- マネージャー：マネジメントを行う

最も重要なのはコミュニケーション方法である。デザイナーとマネージャーは、エンジニア側のコミュニケーション方法を実践しなくてはならない。たとえば GitHub Issues や Cosense、その他ノートやウィキは使えねばならない。もっと言えば非同期コミュニケーション、プレーンテキストベースのコミュニケーションといったものである。もちろんエンジニア自身もこれらができねばならない。ここを解決できない場合、このパターンは形骸化するか、少なくとも一人以上が慢性的な高負荷に陥る。

### Worker、Helper、Advisor パターン
3人

- Worker：実作業を行う
- Helper：Worker を直接支援する。マネジメントも含むが、これに限らない
- Advisor：Worker を改善・変革するための調査や検討を行い、Helper に連携する

このパターンを私は **Worker Triangle** と呼んでいる。

まず一番大事なのが手を動かす Worker であり、この人を最大限尊重します。しかし、Worker だけではまかなえない部分が出てくるので、Helper を置きます。わかりやすく言えば、Worker は直接作業、Helper は間接作業を担います。従来だと全員が直接作業も間接作業もやらされますが、そうではなく、優秀な Worker にだけ任せるイメージです。そのかわり、間接作業は Helper がやる。

次に、これだけだと改善や変革ができないので、もう一つ据えます。Advisor です。その名のとおり助言を行うのですが、対象は Worker ではなく Helper です。というのも、Worker は直接手を動かすことに特化しているので、改善や変革といった目線には弱いからです（エンジニアがソフトスキルを嫌うのと一緒です）。しかし、それでは何もできないので Helper を経由させるわけです。

```
 Worker <--> Helper <--> Advisor
```

そういうわけで、一番重要なのは Helper だったりします。Worker の事情を理解して Advisor に伝えるのはもちろん、Advisor からの先進的な提案も理解した上で、それを Worker に伝えなければならないからです。そういう意味では、Helper ではなく Transolator と呼んでもいいかもしれません。
