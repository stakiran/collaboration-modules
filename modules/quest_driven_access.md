# Quest-Driven Access
**QDA(Quest-Driven Access)** とは、クエスト（何らかの操作実績）をこなした者に段階的に権限を付与していくという権限付与モデルのこと。特に全社員や全メンバーなど広域に権限を付与したい場合に使える、バランスの良いソリューションである。

従来は以下の二通りしかなかった:

- ホワイトリスト方式。必要な者が個別に承認をして手に入れる方式である。これは申請側も管理側もコストがかかるし、承認側の承認判断が通常融通が利きずらく、またボトルネックにもなりがちである。
- オール方式。全員に一気に与える方式である。通常、これは現実的ではない。ありえるとすれば MS365 や Slack といったものであり、以下条件をすべて満たす必要がある
    - 1: すでに広く知られていること
    - 2: 業務の根幹を支えるツールであること（コミュニケーションやストレージなど）
    - 3: エンタープライスレベルのガバナンスとコンプライアンスを満たせること
    - 4: 現実的に安価であること。あるいは顧客ごとに個別に調整できること

QDA であれば、まずは廉価で最低限のアカウントを全員に付与しつつ、実際に多用する者はクエストをクリアしていくので権限を開放していける。ホワイトリスト方式でもなく、オール方式でもない、バランスの良さを実現できる。またクエストのログとアナリティクスを取れば、利用状況も把握できる。

## 例
QDA を実装した例は 2026/01/26 現在、皆無に等しいだろうが、部分的には珍しくない:

- Discord では段階的にロールを付与できるようになっている
- OpenAI API は Tier を敷いており、API 使用量に応じて利用上限が上がっていく
- ゲームでは完全にオープンワールドにはせず、想定ルートを順に辿っていけるよう徐々にダンジョンやアイテムやスキルを解禁していく設計にすることが多い
